# SP29 — Collapse the risk-tier scale to 0–3

## Problem / decision

The five-tier scale (0–4) carries **two verification-redundant pairs**: on the
independence axis there are only three states (self-check / fresh-bundle /
cross-model+adversarial), but five tiers. Old 0 and 1 both self-check; old 2 and
3 both fresh-bundle. The seams differ only in *route*, and the weakest is 0↔1:

- Under a PR/merge obligation (the hosted default — no direct commits to main,
  `commits.md`), old Tier 0's defining "direct commit" is largely fictional. A
  no-behavior typo fix still incurs branch → commit → PR → `process-gates` CI →
  merge. The *only* thing Tier 0 saved over Tier 1 was the one-line risk
  statement. That is not a tier's worth of distinction.

Decision: fold old-0 into old-1 and renumber to a zero-based **0–3** scale.
Chosen over the alternative (fold old-2 into old-3) because the 2↔3 route jump
(Quick vs full Plan→Execute→Review, plus the contract/user-visible scope trigger)
is a *large*, load-bearing distinction we keep; the 0↔1 route jump is the small
one.

## Mapping (formula: `new = max(0, old - 1)`)

| new | old | Scope | Verification |
|---|---|---|---|
| **0** | 0 + 1 | No behavior change **or** local/isolated/reversible | in-context self-check |
| **1** | 2 | Small feature/fix, no contract/persistence/auth impact | fresh-bundle review |
| **2** | 3 | User-visible, cross-component, or interface/contract | fresh-bundle review + Plan cycle |
| **3** | 4 | Auth, persistence/migrations, security surface, multi-repo | cross-model + adversarial + upfront design |

Every boundary now changes at least one axis: 0→1 flips independence
(self→bundle) and adds the test; 1→2 is the (kept, deliberate) route/scope seam
(Quick→Plan, contract trigger); 2→3 escalates independence (bundle→cross-model).

## Gate threshold remap (correctness-critical)

- `check_review.py`: presence enforcement `tier < 3` → `tier < 2`; arithmetic
  `tier >= 2` (bundle/non-implementing required) → `tier >= 1`; cross-model
  `tier == 4` → `tier == 3`. Docstring `N >= 3` → `N >= 2`.
- `check_issues.py`: issue-before-code enforced at old `tier >= 3` → new
  `tier >= 2` (the `int(m.group(1)) < 3` guard → `< 2`).

## verification-independence.md band remap

self-check `Tier 0–1` → `Tier 0`; fresh-bundle `Tier 2–3` → `Tier 1–2`;
cross-model `Tier 4` → `Tier 3`.

## Scope

Template only (the shipped product) + tests. Historical `docs/plans/*` and
`docs/design/*` in this repo keep their original numbers — they record what the
process *was* at that date, not what it *is*. No adopter has real archived-plan
tier data against the old scale (this is the template); tests define their own.

## Anchor Delta

Tier matrix reshaped (SSOT `risk-tiers.md`); two gate thresholds shifted; all
anchor/command/module docs that route by tier number remapped. No new gate, no
new module, no rule added — a scale change, not a capability change.

## Feature Registry Trace

Template self-change; template tests are acceptance. `test_review.py` /
`test_github_issues.py` tier literals remap to the new scale; the presence,
arithmetic, and issue-before-code thresholds are re-proven at the new numbers.

## Non-goals

- Not touching the 2↔3 (new 1↔2) route seam — it stays as the one deliberate
  route-only boundary; its scope trigger is the most-used escalation in practice.
- Not adding a Tier "4"/5 — the cross-model+adversarial+design ceiling is the top
  review state; anything "bigger" is a deployment concern (canary/staged/sign-off),
  not a review tier.
